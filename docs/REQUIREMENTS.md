# 요구사항 명세서 (REQUIREMENTS) — Disaster Alert Archive

> 상위 제품 목표는 [PRD.md](./PRD.md), 기술 설계는 [TRD.md](./TRD.md) 참고. 이 문서는 각 기능을 검증 가능한 단위 요구사항으로 분해한다.

## 1. 기능 요구사항 (Functional Requirements)

| ID | 요구사항 | 우선순위 | 비고 |
|---|---|---|---|
| FR-01 | 시스템은 행정안전부 공공 API로부터 재난 안전 문자를 주기적으로 수집해 DB에 저장해야 한다 | High | 스케줄러 기반 수집 |
| FR-02 | 사용자는 지역·기간·재난 유형으로 재난 문자를 검색/필터링할 수 있어야 한다 | High | |
| FR-03 | 시스템은 동일 사건에 대해 발송된 다건의 재난 문자를 하나의 이벤트로 자동 병합해야 한다 | High | 임베딩 유사도 + LLM 판정 |
| FR-04 | 시스템은 지역을 이동하는 사건(실종자, 탈출 동물 등)에 대해 지역 간 이벤트를 교차 병합할 수 있어야 한다 | Medium | |
| FR-05 | 시스템은 진행 중인 동일 사건에 대한 반복 알림을 쿨다운 처리해 중복 발송을 방지해야 한다 | High | 계절성 안전안내 문자는 쿨다운 갱신 대상에서 제외 |
| FR-06 | 사용자는 회원가입 없이 관심 지역만 등록해 게스트로 푸시 알림을 받을 수 있어야 한다 | High | |
| FR-07 | 사용자는 시/도 단위로 관심 지역을 "전체" 등록할 수 있어야 한다 | Medium | 구 단위로 세분화하지 않아도 시도 전체 알림 수신 |
| FR-08 | 사용자는 알림 유형별로 수신 여부(푸시/미표시/끄기)를 개별 설정할 수 있어야 한다 | Medium | |
| FR-09 | 사용자는 본인이 수신한 알림 이력을 조회할 수 있어야 한다 | Low | |
| FR-10 | 시스템은 재난 유형별 가중치·강도·시간 감쇠를 반영해 지역별 위험도 지수를 산출해야 한다 | Medium | |
| FR-11 | 사용자는 지역별·유형별·기간별 재난 통계를 대시보드에서 확인할 수 있어야 한다 | Medium | |
| FR-12 | 사용자는 기상 관측 데이터와 재난 발생의 상관 관계를 조회할 수 있어야 한다 | Low | |
| FR-13 | 사용자는 재난 현장에 대한 제보(글)를 작성할 수 있어야 한다 | Medium | |
| FR-14 | 사용자는 재난 문자별로 댓글을 작성할 수 있어야 한다 | Medium | |
| FR-15 | 관리자는 허위·장난성 제보/댓글을 검토해 숨김 처리할 수 있어야 한다 | High | 신규 |
| FR-16 | 관리자는 허위·장난성 콘텐츠를 반복 작성한 사용자 계정에 경고 또는 이용 정지(일시/영구)를 적용할 수 있어야 한다 | High | 신규 |
| FR-17 | 이용 정지된 계정은 제보/댓글 작성이 차단되어야 한다 | High | 신규, FR-16과 연동 |
| FR-18 | 시스템은 실종 관련 재난 문자에서 추출한 정보로 실종자를 조회할 수 있게 해야 한다 | Medium | 외부 실종자 API 미연동, 재난 문자만으로 추적 |
| FR-19 | 시스템은 재난 문자와 법정동 명칭을 한국어/영어/중국어/일본어로 번역해 제공해야 한다 | Medium | |
| FR-20 | 사용자는 이메일 또는 Google/Kakao/Naver 소셜 계정으로 회원가입·로그인할 수 있어야 한다 | High | |
| FR-21 | 외부 개발자는 토큰 인증을 거쳐 재난 문자 데이터를 JSON/CSV로 제공받을 수 있어야 한다 | Low | 공개 OpenAPI |

## 2. 비기능 요구사항 (Non-Functional Requirements)

| ID | 요구사항 | 우선순위 | 비고 |
|---|---|---|---|
| NFR-01 | 재난 문자 수집~알림 발송까지의 지연은 최소화되어야 한다 | High | 수집 스케줄러 주기, LLM 호출량에 의존 |
| NFR-02 | 동일 사건에 대한 중복 알림 발송률은 최소화되어야 한다 | High | 클러스터링 + 쿨다운으로 대응 |
| NFR-03 | 만료/무효화된 FCM 토큰으로의 발송 시도를 최소화해야 한다 | Medium | 현재 만료 감지는 되나 자동 삭제는 미구현 — 개선 필요 |
| NFR-04 | 번역 실패 시 원문이 유실되지 않고 그대로 노출되어야 한다 | Medium | |
| NFR-05 | 인증 토큰(JWT)은 탈취 위험을 낮추기 위해 httpOnly 쿠키로만 전달되어야 한다 | High | |
| NFR-06 | DB 스키마 변경은 항상 마이그레이션 파일로 관리되어 재현 가능해야 한다 | High | Flyway, `ddl-auto: validate` |
| NFR-07 | 관리자 전용 기능(제재 등)은 일반 사용자 권한으로 접근할 수 없어야 한다 | High | 신규, `MemberRole.ADMIN` 기반 인가 |
