# PRD (Product Requirements Document) — Disaster Alert Archive

## 1. 개요

재난 안전 문자(공공 재난문자)를 실시간으로 수집·아카이빙하고, 사용자가 등록한 관심 지역을 기준으로 맞춤형 푸시 알림을 제공하는 서비스. 단순 문자 중계를 넘어, 동일 사건에 대한 중복 알림을 AI 기반으로 클러스터링해 하나의 "이벤트"로 묶고, 지역별 위험도를 정량화해 제공한다.

- **배포 URL**: https://disaster-alert-archive.co.kr/

## 2. 배경 및 문제 정의

행정안전부의 재난문자는 발령 기관·행정구역 단위가 세분화되어 있어, 동일한 하나의 재난(예: 산불, 붕괴 사고)이 여러 건의 개별 문자로 중복 발송되는 경우가 많다. 사용자는 이 때문에 같은 사건에 대해 반복 알림을 받거나, 정작 자신이 관심 있는 지역의 정보를 놓치기 쉽다. 또한 재난 문자는 발생 시점의 단발 정보만 제공할 뿐, 특정 지역이 시간에 따라 얼마나 위험한 상태인지 누적된 형태로 보여주지 않는다.

## 3. 목표

1. 관심 지역 기반으로 필요한 재난 알림만 정확히 받게 한다.
2. 동일 사건에 대한 중복 알림을 제거해 알림 피로도를 낮춘다.
3. 과거 재난 이력과 지역별 위험도를 데이터로 확인할 수 있게 한다.
4. 외국인 거주자·관광객을 포함해 다국어로 재난 정보에 접근할 수 있게 한다.

## 4. 타겟 사용자

- 특정 지역(거주지, 근무지, 가족 거주지 등)의 재난 상황을 실시간으로 확인하고 싶은 일반 사용자
- 한국어가 모국어가 아닌 거주자·체류자 (다국어 지원 대상: 한/영/중/일)
- 비로그인 상태로 관심 지역 알림만 받고 싶은 게스트 사용자 (게스트 FCM 지원)
- 재난 데이터를 외부 시스템에서 활용하려는 개발자 (공개 OpenAPI 대상)

## 5. 핵심 기능

### 5.1 재난 문자 아카이빙
- 행정안전부 공공 API를 스케줄러로 주기 수집, DB 저장
- 지역·기간·재난 유형 등으로 검색/필터

### 5.2 이벤트 클러스터링 (AI)
- 임베딩(텍스트 임베딩) + LLM 판정으로 동일 사건에 대한 다건의 알림을 하나의 이벤트로 병합
- 지역 이동형 사건(실종자, 탈출 동물 등)에 대한 지역 간 교차 병합
- 파편화된 이벤트를 주기적으로 재결합하는 백그라운드 정합화

### 5.3 맞춤 푸시 알림
- 회원가입 없이도 관심 지역만 등록해 게스트로 푸시 수신 가능
- 시/도 단위 "전체" 등록 지원 (구 단위로 세분화하지 않아도 됨)
- 알림 유형별(PUSH/알림만/끄기) 개인화 설정
- 알림 수신 이력 조회

### 5.4 지역별 위험도 분석
- 재난 유형별 가중치 × 강도 × 시간 경과에 따른 감쇠를 반영한 지역 위험도 지수 산출
- 지도 기반 시각화

### 5.5 재난 통계 대시보드
- 지역별·유형별·기간별 통계
- 기상 관측 데이터와의 상관 분석 (기상청 데이터 연계)

### 5.6 커뮤니티
- 사용자 재난 제보
- 재난 문자별 댓글

### 5.7 사용자 제재 (신규 요구사항, 관리자)
- 허위·장난성 커스텀 제보(재난 제보/댓글)를 관리자가 검토해 제재할 수 있어야 함
- 제재 대상 콘텐츠 삭제(숨김) 및 작성자 계정에 대한 경고/이용 정지(일시·영구) 처리
- 정지된 계정은 제보/댓글 작성이 제한되어야 함
- 관리자(`MemberRole.ADMIN`) 권한 기반으로 동작 — 현재 회원 도메인에 `USER`/`ADMIN` 역할 구분은 있으나, 계정 정지 상태 필드와 관리자용 제재 화면/API는 아직 없어 신규 구현 필요

### 5.8 실종자 정보
- 별도의 외부 실종자 API(경찰청 등) 연계 없이, 재난 안전 문자 중 실종 관련 문자에서 추출한 정보만으로 실종자를 추적·조회

### 5.9 다국어 지원
- 한국어/영어/중국어/일본어 번역 (DeepL API 기반, 재난 문자 및 법정동 명칭 포함)

### 5.10 계정 및 인증
- 이메일 회원가입/로그인
- Google, Kakao, Naver 소셜 로그인

### 5.11 공개 API
- 토큰 기반 재난 문자 공개 API (JSON/CSV), 외부 개발자 활용 목적

## 6. 비기능 요구사항

- 사용자에게 전달되는 알림은 실제 사건 발생 시점 대비 지연이 최소화되어야 한다 (수집 스케줄러 주기에 의존).
- 동일 사건에 대한 중복 알림 발송을 방지해야 한다 (쿨다운 메커니즘).
- 재난 상황이라는 서비스 특성상 알림 발송 실패(만료된 토큰 등)를 최소화해야 한다.
- 다국어 번역은 원문 재난 문자의 의미를 왜곡하지 않아야 한다.

## 7. 범위 외 (Out of Scope)

- 재난 상황에 대한 신고·구조 요청 등 긴급 대응 기능 (본 서비스는 정보 제공·아카이빙 목적)
- 재난문자를 발령하는 기능 자체 (발령 주체는 행정안전부 등 공공 기관)
