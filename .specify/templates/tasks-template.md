# 작업 목록: [FEATURE NAME]

**입력**: `/specs/[###-feature-name]/`의 설계 문서

**전제조건**: plan.md(필수), spec.md(사용자 스토리가 있는 경우 필수), research.md, data-model.md, contracts/

**테스트**: 아래 예시에는 테스트 작업이 포함되어 있습니다. 테스트는 선택사항이며, 기능 명세에서 명시적으로 요청한 경우에만 포함하세요.

**구성**: 작업은 각 스토리를 독립적으로 구현/테스트할 수 있도록 사용자 스토리별로 그룹화됩니다.

## 형식: `[ID] [P?] [Story] Description`

- **[P]**: 병렬 실행 가능 (다른 파일, 의존성 없음)
- **[Story]**: 이 작업이 속한 사용자 스토리 (예: US1, US2, US3)
- 설명에 정확한 파일 경로를 포함할 것

## 경로 규칙

- **Single project**: 저장소 루트의 `src/`, `tests/`
- **Web app**: `backend/src/`, `frontend/src/`
- **Mobile**: `api/src/`, `ios/src/` 또는 `android/src/`
- 아래 경로는 single project를 가정합니다 - plan.md 구조에 맞게 조정하세요

## 의존성 및 실행 순서

### Phase 의존성

- **설정 (Phase 1)**: 의존성 없음 - 즉시 시작 가능
- **기반 작업 (Phase 2)**: 설정 완료에 의존 - 모든 사용자 스토리를 BLOCK함
- **사용자 스토리 (Phase 3+)**: 모두 기반 작업 phase 완료에 의존
  - 이후 사용자 스토리는 (인력이 있다면) 병렬로 진행 가능
  - 또는 우선순위 순서대로 순차 진행 (P1 → P2 → P3)
- **마무리 (마지막 Phase)**: 원하는 모든 사용자 스토리가 완료됨에 의존

### 사용자 스토리 의존성

- **사용자 스토리 1 (P1)**: 기반 작업(Phase 2) 이후 시작 가능 - 다른 스토리에 대한 의존성 없음
- **사용자 스토리 2 (P2)**: 기반 작업(Phase 2) 이후 시작 가능 - US1과 통합될 수 있으나 독립적으로 테스트 가능해야 함
- **사용자 스토리 3 (P3)**: 기반 작업(Phase 2) 이후 시작 가능 - US1/US2와 통합될 수 있으나 독립적으로 테스트 가능해야 함

### 각 사용자 스토리 내부

- 테스트(포함된 경우)는 반드시 구현 전에 작성되고 FAIL해야 함
- 모델이 서비스보다 먼저
- 서비스가 엔드포인트보다 먼저
- 핵심 구현이 통합보다 먼저
- 스토리를 완료한 뒤 다음 우선순위로 이동

### 병렬 처리 가능 지점

- [P] 표시된 모든 설정 작업은 병렬 실행 가능
- [P] 표시된 모든 기반 작업은 (Phase 2 내에서) 병렬 실행 가능
- 기반 작업 phase가 완료되면 (팀 여력이 되는 한) 모든 사용자 스토리를 병렬로 시작 가능
- [P] 표시된 한 사용자 스토리의 모든 테스트는 병렬 실행 가능
- [P] 표시된 한 스토리 내 모델들은 병렬 실행 가능
- 서로 다른 사용자 스토리는 다른 팀원이 병렬로 작업 가능

## 구현 전략

### MVP 우선 (사용자 스토리 1만)

1. Phase 1: 설정 완료
2. Phase 2: 기반 작업 완료 (중요 - 모든 스토리를 BLOCK함)
3. Phase 3: 사용자 스토리 1 완료
4. **중지 및 검증**: 사용자 스토리 1을 독립적으로 테스트
5. 준비되면 배포/데모

### 점진적 전달

1. 설정 + 기반 작업 완료 → 기반 준비 완료
2. 사용자 스토리 1 추가 → 독립 테스트 → 배포/데모 (MVP!)
3. 사용자 스토리 2 추가 → 독립 테스트 → 배포/데모
4. 사용자 스토리 3 추가 → 독립 테스트 → 배포/데모
5. 각 스토리는 이전 스토리를 깨뜨리지 않고 가치를 더함

### 병렬 팀 전략

여러 개발자가 있는 경우:

1. 팀이 함께 설정 + 기반 작업 완료
2. 기반 작업이 끝나면:
   - 개발자 A: 사용자 스토리 1
   - 개발자 B: 사용자 스토리 2
   - 개발자 C: 사용자 스토리 3
3. 스토리들이 독립적으로 완료 및 통합됨

## 참고

- [P] 작업 = 서로 다른 파일, 의존성 없음
- [Story] 라벨은 작업을 특정 사용자 스토리에 매핑하여 추적 가능하게 함
- 각 사용자 스토리는 독립적으로 완료 및 테스트 가능해야 함
- 구현 전에 테스트가 FAIL하는지 확인할 것
- 각 작업 또는 논리적 그룹 단위로 커밋할 것
- 스토리를 독립적으로 검증하기 위해 어떤 체크포인트에서든 멈출 수 있음
- 피할 것: 모호한 작업, 동일 파일 충돌, 독립성을 깨뜨리는 스토리 간 의존성
